# Integration: Flutter + Amplify Monorepo

Adding Flutter mobile support to an Amplify web project requires converting to a monorepo structure. Both web and mobile share the same Amplify backend, which means changes to the backend schema or auth configuration affect both platforms simultaneously.

## Convert to Monorepo First

The monorepo conversion **must** happen before adding Flutter. Attempting to add Flutter to a flat project structure creates conflicts between npm and Flutter package management.

### Target Structure

```
project-root/
├── amplify/                    # Shared backend (unchanged location)
│   ├── auth/
│   ├── data/
│   ├── backend.ts
│   └── package.json
├── packages/
│   ├── web/                    # Existing Next.js app (moved here)
│   │   ├── src/
│   │   ├── package.json
│   │   └── next.config.ts
│   └── mobile/                 # New Flutter app
│       ├── lib/
│       ├── assets/
│       │   └── amplify_outputs.json  # Copied from root
│       └── pubspec.yaml
├── package.json                # Root with workspaces
├── amplify_outputs.json        # Generated by Amplify
└── node_modules/
```

### Root package.json Workspaces

The root `package.json` must declare npm workspaces:

```json
{
  "name": "project-root",
  "private": true,
  "workspaces": [
    "packages/*"
  ]
}
```

This allows npm to hoist shared dependencies and manage both the root Amplify tooling and the web package together.

## Shared Amplify Backend

The `amplify/` directory remains at the project root and serves both web and mobile. Key implications:

- **Schema changes** (in `amplify/data/resource.ts`) affect both platforms
- **Auth changes** (in `amplify/auth/resource.ts`) affect both platforms
- **Deployments** rebuild the entire backend regardless of which platform triggered the change
- Both platforms consume the same `amplify_outputs.json` for configuration

## Flutter Prerequisites

Flutter can only be added **after Layer 1** (Amplify infrastructure) is complete:

1. Amplify backend must be deployed
2. `amplify_outputs.json` must be generated (contains AppSync endpoint, Cognito config, etc.)
3. AppSync endpoint must be working and accessible

Without a deployed backend, Flutter has nothing to connect to and `amplify_outputs.json` does not exist.

## Config Sync: amplify_outputs.json

Amplify generates `amplify_outputs.json` at the project root. Flutter needs this file in its assets directory.

```
amplify_outputs.json (root, generated by Amplify)
        │
        └──► packages/mobile/assets/amplify_outputs.json (copied)
```

This copy must be kept in sync. When the backend changes (new model, auth config update), regenerate `amplify_outputs.json` and copy it to the mobile assets directory.

Consider a script or npm lifecycle hook to automate this:

```json
{
  "scripts": {
    "amplify:sync": "cp amplify_outputs.json packages/mobile/assets/amplify_outputs.json"
  }
}
```

## Two Package Managers Coexist

The monorepo uses two separate package management systems:

| Platform | Package Manager | Config File | Lock File |
|----------|----------------|-------------|-----------|
| Web (Next.js) | npm | `package.json` | `package-lock.json` |
| Mobile (Flutter) | pub | `pubspec.yaml` | `pubspec.lock` |

These do not conflict because they operate in separate directories and manage different dependency ecosystems. However, CI/CD pipelines must install dependencies for both:

```bash
# Web dependencies
npm install

# Mobile dependencies
cd packages/mobile && flutter pub get
```

## Port Allocation

Web and mobile development servers, plus design tooling, all need separate ports. Allocate ports explicitly to avoid conflicts:

| Service | Purpose | Example Port |
|---------|---------|-------------|
| Next.js dev server | Web development | 3000 |
| Storybook | Web component development | 6006 |
| Widgetbook | Flutter widget development | 49152 |
| Flutter dev server | Mobile development (web target) | 8080 |

Document allocated ports in `PORT_INFO.md` at the project root to prevent collisions.

## Migration Checklist

When converting an existing web project to a monorepo with Flutter:

- [ ] Create `packages/` directory
- [ ] Move web app to `packages/web/`
- [ ] Update root `package.json` with `"workspaces": ["packages/*"]`
- [ ] Verify `amplify/` stays at project root (do not move)
- [ ] Update any absolute import paths in the web app
- [ ] Update CI/CD paths (build commands, working directories)
- [ ] Update Amplify build settings (`appRoot` if needed)
- [ ] Create Flutter app in `packages/mobile/`
- [ ] Copy `amplify_outputs.json` to `packages/mobile/assets/`
- [ ] Configure port allocation in `PORT_INFO.md`
- [ ] Verify web app still builds and deploys correctly
- [ ] Verify Flutter app connects to Amplify backend
